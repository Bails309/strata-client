# ADR-0012: NJS-based Security Hardening (CSP & Header Masking)

## Status

Accepted

## Context

Security audits and technology fingerprinting tools can often identify underlying infrastructure by inspecting HTTP headers (e.g., `Server: nginx/1.25.3`). Additionally, modern web security requires robust protection against clickjacking via `Content-Security-Policy`. While the application backend can set these headers, certain responses (like 404s or 50x errors generated by Nginx itself) might bypass the backend's header logic.

## Decision

Implement a defense-in-depth layer using the **Nginx JavaScript (NJS)** module within the `frontend` container.

1.  **Header Masking**: Use NJS to intercept every response and rewrite the `Server` header to "Strata".
2.  **Removal of Sensitive Headers**: Automatically strip `X-Powered-By` and other infrastructure-revealing headers.
3.  **Global CSP Enforcement**: Apply `Content-Security-Policy: frame-ancestors 'none'` at the gateway level. This ensures that even static assets and Nginx-generated error pages are protected against clickjacking.
4.  **Decoupling**: By handling these headers at the Nginx level, the application backend remains focused on business logic, while the gateway ensures a consistent security posture.

## Consequences

- **Reduced Fingerprinting**: Attackers have less information about the specific Nginx version or backend technology being used.
- **Robust Clickjacking Protection**: The `frame-ancestors 'none'` policy is enforced globally, providing superior protection over the legacy `X-Frame-Options`.
- **Improved Compliance**: The system more easily meets strict security compliance requirements (e.g., OWASP Top 10, SOC2) by ensuring all responses carry required security headers.
