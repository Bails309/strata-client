# ADR-0006 ‚Äî Vault Transit envelope encryption for PM credentials

- **Status**: Accepted
- **Date**: 2026-04-21
- **Wave**: W5-12
- **Related standards**: ¬ß4.5 (secrets at rest), ¬ß4.6 (key rotation), ¬ß15 (trust boundaries)
- **Supersedes**: ‚Äî
- **Superseded by**: ‚Äî

## Context

Strata stores several classes of secret at rest in PostgreSQL:

* SSO client secrets
* LDAP / AD bind passwords
* Password-Manager (PM) bind passwords and managed-account credentials
* Kerberos keytabs (binary)
* Azure storage access keys for session recordings

Coding Standards ¬ß4.5 forbids storing plaintext secrets in any
database row, and ¬ß4.6 requires a rotation path that does not require
rewriting every ciphertext blob.

HashiCorp Vault's **Transit** secrets engine provides envelope
encryption that meets both requirements without Strata ever holding
a long-lived data-encryption key (DEK) in memory longer than the
duration of a single request.

## Decision

**Accept Vault Transit envelope encryption as the single at-rest
encryption mechanism for all operator-managed secrets.** Plaintext
secrets MUST NOT be written to `settings`, `ad_sync_configs`, or any
other table. Any existing plaintext column is migrated on read and
re-written as an envelope on the next write.

### Envelope shape

The on-disk value for any sealed secret is the literal string:

```
vault:<base64-of-vault-transit-ciphertext>
```

where `<vault-transit-ciphertext>` is the exact payload returned by
Vault's `transit/encrypt/<key>` endpoint (which itself is of the
form `vault:v<N>:<opaque>` ‚Äî we wrap it a second time in base64 for
DB-safety and to avoid parser ambiguity at read time).

Reader contract: if a setting value starts with `"vault:"`, call
`crate::services::vault::unseal_setting` to decrypt; otherwise treat
as plaintext (legacy / non-secret).

Writer contract: any write path for a secret column MUST call
`crate::services::vault::seal` (or `seal_setting`) before the SQL
UPDATE. Examples:

* `SSO client secret` ‚Äî [backend/src/routes/admin.rs](backend/src/routes/admin.rs)
  seals the client secret on submit and stores `"vault:..."` in
  `settings.sso_client_secret`.
* `AD bind password` ‚Äî same path for `ad_sync_configs.bind_password`.
* `PM bind password` and `PM target passwords` ‚Äî same path via
  `pm_bind_password` and the checkout-issue flow.

### Key hierarchy

| Layer | What it does | Where it lives |
|---|---|---|
| **Vault root / unseal keys** | Bootstrap Vault itself | Operator-held, split via Shamir |
| **KEK ‚Äî Transit key `guac-master-key`** | The named Transit key that wraps every Strata ciphertext | Vault, never exits the Vault process |
| **DEK ‚Äî per-encryption data key** | Ephemeral; generated by Vault per `encrypt` call | Never materialised in Strata; Vault emits ciphertext only |
| **Ciphertext** | `vault:<b64>` as above | Postgres row |

Strata never sees a DEK. Every read is a round-trip to Vault, which
is accepted because:

* The secret-touching paths are all admin-config changes, login
  paths, and session-open paths ‚Äî not hot-loop user traffic.
* Vault is on the same Docker network as the backend
  (`strata-internal`, see ADR-0004 for analogous reasoning).
* Client-side caching of decrypted secrets is limited to the
  lifetime of a single request future.

### Rotation

Coding Standards ¬ß4.6 requires key rotation without mass-rewriting
ciphertext. Vault Transit supports this natively:

1. Operator runs `vault write -f transit/keys/guac-master-key/rotate`.
2. New writes use the new key version (`vault:v2:...`).
3. Existing ciphertexts continue to decrypt because Vault keeps
   historical versions.
4. **Rewrap** (not re-encrypt) endpoint is called by a scheduled
   admin-only task to promote existing ciphertexts to the newest
   version without exposing plaintext to Strata.

This means a compromised DEK is bounded to the single row it
encrypted, and rotating the KEK does not require touching
application data.

## Consequences

**Positive**

* Zero long-lived key material in the Strata process image.
* A Postgres dump is not useful to an attacker without also
  compromising Vault's unseal keys and the network path between
  backend and Vault.
* Rotation is a Vault operation, not a database migration.
* Legacy plaintext rows and envelope rows can coexist during
  migration because the `"vault:"` prefix is self-identifying.

**Negative**

* Hard dependency on Vault being reachable on the secret-touching
  code paths. A Vault outage degrades login, SSO config, AD sync,
  and PM checkout. Mitigations:
  * Health check exposes Vault status and the admin UI surfaces it.
  * Non-secret read paths (ordinary API traffic) never call Vault.
* Every secret read is an RTT to Vault ‚Äî accepted because these
  paths are not on the high-QPS request surface.
* Local-dev vault mode uses a file-backed stub so that contributors
  without a Vault service can still run the app; this fallback is
  documented and gated by config so it can never be enabled in
  production builds.
